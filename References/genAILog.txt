## Quick summary
The `/profile` route in `index.js` has a few bugs that prevent it from correctly loading and rendering a user's profile:
- a SQL column name typo,
- an improper DB call / returned shape vs. how the template expects data,
- incomplete error logging in the catch block.

Because of these, the query will likely return no rows (even for an existing user) or return an unexpected shape, and the template `src/views/pages/profile.hbs` will not have the user fields available at the top-level context.

## Issues found (details and impact)
1. SQL column name typo
   - Current WHERE clause: `WHERE usename = $1`
   - Problem: `usename` is misspelled (missing the 'r'). The `users` table uses `username`.
   - Impact: The DB will return no rows or an SQL error depending on DB setup; the user record won't be retrieved.

2. DB helper used and returned data shape
   - Current code uses `db.any(...)`. `db.any` always returns an array (possibly empty).
   - Later the code does: `res.render('pages/profile', {userInfo});`
     - That passes an object with one property `userInfo` whose value is an array.
     - The template `profile.hbs` expects top-level fields (`{{this.username}}`, `{{this.phone_num}}`, `{{this.email}}`) — i.e., it expects the context to be the user object itself.
   - Impact: Template won't find `username` at top-level and will render nothing for those fields.

3. Error logging in catch
   - Current catch block: `catch (err) { console.error(); }`
   - Problem: `console.error()` is called with no argument, so the actual error is not logged.
   - Impact: Harder to debug runtime errors.

4. (Behavioral) Global auth middleware
   - `app.use(auth)` is declared before `/profile`, so access requires a logged-in session. That's fine if intentional, but verify you want auth applied globally to later routes.

## Recommended fix (small, focused)
- Fix the SQL to use `username`.
- Use `db.oneOrNone(...)` (since username is unique) to get a single object or null.
- If no user found, render an error or redirect.
- Pass the user object as the top-level render context so `{{this.username}}` works.
- Fix the catch logging to include the error.

Replace the current `/profile` handler with the following:

```javascript
app.get('/profile', async (req, res) => {
  try {
    const userInfo = await db.oneOrNone(
      `SELECT username, email, phone_num
       FROM users
       WHERE username = $1`,
      [req.session.user.username]
    );

    if (!userInfo) {
      // no user found — handle gracefully
      return res.status(404).render('pages/profile', { error: 'User not found.' });
    }

    // pass the user object as the top-level context
    res.render('pages/profile', userInfo);
  } catch (err) {
    console.error('Profile route error:', err);
    res.status(500).render('pages/error', { error: 'Could not load profile.' });
  }
});
```

Why this works:
- `db.oneOrNone` returns either an object (with `username`, `email`, `phone_num`) or `null`.
- Rendering with `res.render('pages/profile', userInfo)` makes the template context equal to the user object, so `{{this.username}}` and the other `{{this.*}}` expressions in `profile.hbs` will find the values.
- Proper error logging and status codes make debugging and UX clearer.

## Alternative approaches
- Keep `db.any` but extract the first row and pass it:  
  const rows = await db.any(...); const userInfo = rows[0]; ... res.render('pages/profile', userInfo);
- Or pass a namespaced object and update the template:
  - Server: `res.render('pages/profile', { user: userInfo });`
  - Template: use `{{user.username}}`, `{{user.email}}`, etc.
- If you prefer explicit fields in the template, pass `{ username: userInfo.username, email: userInfo.email, phone_num: userInfo.phone_num }`.

## Edge cases and suggestions
- If `req.session.user` can be missing despite auth middleware, add a defensive check before DB call.
- Consider using a dedicated `profile` view model object to avoid leaking fields.
- Keep templates escaping in mind (Handlebars escapes by default).
- If you want to reuse the same pattern elsewhere, consider a small helper to fetch the current user.

  const { username, email, phone, verificationCode } = req.body;
  const currentUser = req.session.user;

  if (!username || !email || !phone) {
    return res.status(400).json({ ok: false, msg: 'All fields are required' });
  }

  try {
    const currentUserData = await db.oneOrNone(
      'SELECT email FROM users WHERE user_id = $1',
      [currentUser.user_id]

    const existing = await db.oneOrNone(

      [email, username, currentUser.user_id]
    );
    if (existing) {
      return res.status(400).json({ ok: false, msg: 'Email or username already taken' });
    }

    // Update user in database
    await db.none(
      'UPDATE users SET username = $1, email = $2, phone_num = $3 WHERE user_id = $4',
      [username, email, phone, currentUser.user_id]
    );

    // Update session with new data
    req.session.user.username = username;
    req.session.user.email = email;
    req.session.user.phone_num = phone;
    req.session.emailVerification = null;

    req.session.save(() => {
      res.json({ ok: true, msg: 'Profile updated successfully' });
    });
